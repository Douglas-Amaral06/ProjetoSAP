# 03 — Especificação de Frontend: Interface da Equipe

Escopo: `equipe_app.py`. Depende de `backend.py`, `aiengine.py` e
`theme.py`.

## Objetivo da tela
Painel de trabalho para a equipe de suporte SAP: visualizar e gerenciar
todos os chamados (de todos os usuários), resolver manualmente ou via IA,
manter a base de conhecimento e as normas que orientam a IA, e acompanhar
indicadores de SLA.

## Execução
```bash
python -m streamlit run equipe_app.py --server.port 8502
```
Config da página: `page_title="Painel da Equipe SAP"`, `page_icon="🛠️"`,
`layout="wide"`.

## Tema
Mesmo tema dark de `theme.aplicar_tema_dark()`, compartilhado com a
interface do cliente — qualquer alteração visual deve ser feita em
`theme.py` para valer nas duas telas.

## Abas
`st.tabs(["Fila", "Base de conhecimento", "Normas da IA", "🚨 Alertas", "SLA"])`

## 1. Aba "Fila"

### 1.1 Controles superiores

| Elemento | Tipo | Comportamento |
|---|---|---|
| Filtro de status | `st.selectbox` | `Todos, Aberto, Em Atendimento, Resolvido` |
| Botão "🔄 Atualizar" | `st.button` | `st.rerun()` |
| Auto-refresh | `st_autorefresh(interval=15000, key="refresh_fila")` | Só se `streamlit_autorefresh` estiver instalado |

### 1.2 Tabela
Fonte: `backend.listar_chamados(filtro)` — **todos os usuários**, ao
contrário da tela do cliente. Renderizada com `st.dataframe` mostrando
**todas** as colunas do ticket (não há uma lista restrita de colunas aqui,
diferente da tela do cliente).

Se vazio: `st.info("Nenhum chamado encontrado.")`.

### 1.3 Grid de Cards Clicáveis **[NOVO R8]**
Ao invés de um `number_input` para digitar manualmente o ID, agora há um grid
visual de **cards clicáveis** (3 colunas).

**Cada card mostra:**
- Número do chamado (`#ID`)
- Emoji de prioridade (🟢 Baixa, 🟡 Média, 🟠 Alta, 🔴 Crítica)
- Status do chamado
- Módulo SAP
- Nome do usuário (truncado)
- Primeiras 50 chars da descrição
- Botão "Selecionar #ID" que preenche a seleção automaticamente

**Cores dos cards** (border): baseadas no status (vermelho=Aberto, laranja=Em Atendimento, verde=Resolvido)

### 1.4 Detalhes e Ações do Chamado Selecionado
Após clicar em "Selecionar", a interface mostra:
- Sucesso: "✓ Chamado #X selecionado"
- Seção "Detalhes do Chamado #X" com:
  - 3 métricas: Status, Prioridade, Módulo
  - Descrição completa
  - Solução aplicada (se existir)

#### 1.4.1 Seção de Conversa **[NOVO R7]**

Após os detalhes, exibe uma seção de **comentários/interações**:

- **Listagem de mensagens**: todas as interações do chamado.
  - Mostra contador total: "Total de mensagens: X".
  - Cada mensagem em um container com border.
  - Cada mensagem mostra: emoji/role (👤 Cliente / 👨‍💼 Equipe), username, data, e conteúdo.
  - Ordenadas por data crescente (mais antigas primeiro).
  - Se vazio: `st.info("Nenhuma mensagem ainda.")`.

- **Formulário de nova mensagem**: `st.form` com:
  - `st.text_area` para digitar (altura 80, placeholder "Digite aqui...")
  - Botão "Enviar mensagem".
  - Ao submeter: chama `backend.criar_interacao(ticket_id, usuario_id, mensagem)`.
  - Sucesso: `st.success` + `st.rerun()`.
  - Erro: `st.error(str(erro))`.

#### 1.4.2 Ações

Divisor visual, seguido de seção de **ações disponíveis**:
- **Coluna A — Atualizar status**: selectbox com opções `Aberto, Em Atendimento, Aguardando confirmação` **[NOVO R5]** + botão
- **Coluna B — Resolver com IA**: botão
- **Expander "Resolver manualmente"**: text_area + botão

**[NOVO R5]** Equipe pode mover chamados entre estados intermediários conforme necessário.

### 1.5 Session State
Usa `st.session_state["chamado_selecionado_r8"]` para manter o ID selecionado
entre reloads. Ao completar uma ação (resolver/atualizar), reseta o estado
para forçar a página voltar para "Selecione um chamado acima".

Após listar, equipe seleciona um ID da KB via `st.selectbox`.

## 3. Aba "Normas da IA"

### 3.1 Listagem
Fonte: `backend.listar_todas_as_normas()` (ativas E desativadas, para gestão).
Exibe todas com status visual (ativas normais, desativadas com cor cinza).

### 3.2 Gerenciar Norma
Equipe seleciona uma norma para gerenciar via `st.selectbox`.

Exibe:
- Título, status (🟢 Ativa / 🔴 Desativada), conteúdo

Botões contextuais:
- Se norma está **ativa**: botão "🔴 Desativar norma" → chama `backend.desativar_norma()`.
- Se norma está **desativada**: botão "🟢 Reativar norma" → chama `backend.reativar_norma()`.

**Comportamento**:
- Normas desativadas não entram no prompt do Gemini (via `montar_normas_modelo()` que filtra `ativo = 1`).
- O histórico é mantido — desativar é reversível.

### 3.3 Adicionar Nova Norma

Formulário (`st.form`) com campos:

| Campo | Tipo |
|---|---|
| Título da norma | `st.text_input` |
| Conteúdo da norma | `st.text_area` (altura 120) |
| Botão | `st.form_submit_button("Adicionar norma")` |

Ao submeter: `backend.salvar_norma_modelo(titulo, conteudo)` → `st.success` + `st.rerun()`, ou `st.error(str(erro))`.

**Importante**: toda norma cadastrada aqui **entra automaticamente no prompt do Gemini** na próxima chamada de IA (via `aiengine.montar_normas_modelo()`), sem precisar de deploy — é a forma principal de "ensinar" a IA no piloto.

## 4. Aba "🚨 Alertas" **[NOVO R4]**

Mostra notificações de divergência entre prioridade escolhida pelo cliente e
sugerida pela IA.

### 4.1 Listagem
Fonte: Notificações com `tipo = NOTIF_PRIORIDADE_DIVERGE`.

Se vazio: `st.info("Nenhum alerta de divergência de prioridade...")`.

Se houver dados, cada alerta exibe (em container com border):
- `**Chamado #{ticket_id}**` — com status de leitura (✓ Lido / ● Não lido)
- Status atual do chamado e prioridade atual
- Mensagem completa da divergência (prioridade cliente vs. sugerida + justificativa)
- Data de criação
- Botão "✓ Lido" se ainda não lido (marca como lida e atualiza)

## 5. Aba "SLA"

Fonte: `backend.obter_metricas_sla()` (DataFrame).

Se vazio (`df_sla.empty`): `st.info("Sem dados para calcular SLA.")`.

Se houver dados:

**Métricas em 4 colunas (`st.columns(4)`)**:
1. Total (contagem geral de tickets)
2. Abertos (`status == Aberto`)
3. Em atendimento (`status == Em Atendimento`)
4. SLA estourado (`status_sla == "SLA Estourado"`)

**Métrica isolada abaixo**: Resolvidos (`status == Resolvido`).

**Gráfico**: `st.bar_chart` com a contagem de `status_sla`
(`value_counts()`) — mostra visualmente a proporção "Dentro do SLA" vs
"SLA Estourado".

**Tabela completa**: `st.dataframe(df_sla, ...)` — todas as colunas,
incluindo as derivadas (`status_sla, tempo_resolucao_horas,
horas_ate_prazo`).

## 6. Estados a cobrir (checklist)

- [ ] Fila vazia (nenhum chamado no sistema ainda) → mensagem amigável, sem
      quebrar os controles de filtro.
- [ ] ID digitado no `number_input` que não existe no banco → erro amigável
      ao tentar qualquer ação (status, IA, manual), sem crash.
- [ ] Resolver com IA quando a `GOOGLE_API_KEY` está ausente/inválida →
      erro tratado, chamado permanece no estado anterior (não deve marcar
      como resolvido em caso de falha da chamada à IA).
- [ ] Resolver manualmente com campo de solução vazio → erro de validação
      do backend (`Solução é obrigatório.`), sem gravar nada.
- [ ] Cadastro de norma com título ou conteúdo vazio → erro de validação.
- [ ] Aba SLA sem nenhum ticket resolvido ainda → gráfico e métricas não
      devem quebrar (contagens simplesmente ficam em 0).
