# 01 — Especificação de Backend

Escopo: `backend.py` (dados + regras de negócio) e `aiengine.py` (IA).
Ambos são módulos compartilhados — importados tanto por `cliente_app.py`
quanto por `equipe_app.py`. Qualquer alteração aqui afeta as duas interfaces.

---

## 1. Modelo de dados (SQLite)

### 1.1 Tabela `tickets`

| Coluna | Tipo | Regras |
|---|---|---|

| `usuario` | TEXT NOT NULL | quem abriu o chamado (nome/usuário digitado no cliente) |

| `descricao` | TEXT NOT NULL | texto livre descrevendo o problema |
| `prioridade` | TEXT NOT NULL DEFAULT `'Média'` | um de: `Baixa, Média, Alta, Crítica` |

`init_db()` também roda checagens de `ALTER TABLE ... ADD COLUMN` de forma
idempotente (via `coluna_existe`) para as colunas `prioridade` e
`origem_solucao`, cobrindo bancos criados por versões anteriores do schema.
Qualquer nova coluna futura deve seguir esse mesmo padrão (checar existência
antes de alterar).

## 2. Configuração / variáveis de ambiente

| Variável | Obrigatória | Default | Descrição |
|---|---|---|---|
| `GOOGLE_API_KEY` | Sim (para IA funcionar) | — | chave da API do Gemini |
| `GEMINI_MODEL` | Não | `gemini-2.5-flash` | nome do modelo Gemini usado |
| `SAP_HELPDESK_DB` | Não | caminho ao lado de `backend.py` | caminho do arquivo SQLite |

`backend.py` resolve `DB_NAME` com base em `Path(__file__).with_name(...)`,
ou seja, o caminho do banco é relativo à localização do **módulo**
`backend.py`, não do app que o importa — é isso que garante que
`cliente_app.py` e `equipe_app.py`, mesmo rodando como processos separados,
apontem para o mesmo arquivo `.db`.

## 3. Regras de negócio

### 3.1 Normalização de texto (`normalizar_texto`)
Usada para gerar a chave de deduplicação da base de conhecimento: remove
espaços extras nas pontas, colapsa espaços internos múltiplos em um só, e
converte tudo para minúsculas. **Não** faz remoção de acentos, stemming, nem
qualquer forma de similaridade semântica — é um match exato pós-normalização.

### 3.2 Cálculo de SLA (`calcular_horas_sla`)

| Prioridade | Prazo (horas) |
|---|---|
| Baixa | 72 |
| Média | 24 |
| Alta | 8 |
| Crítica | 4 |

Qualquer prioridade não reconhecida cai no default de 24h. O prazo é gravado
como timestamp absoluto (`data_criacao + horas`) no momento da criação do
chamado, não recalculado depois.

### 3.3 Máquina de estados do `status`
Estados possíveis: `Aberto → Em Atendimento → Resolvido`.

- A transição para `Resolvido` só acontece via `resolver_chamado()` (que
  exige uma solução não vazia) — nunca via `atualizar_status_chamado()`.
- `atualizar_status_chamado()` aceita apenas `Aberto` ou `Em Atendimento`
  como destino; qualquer outro valor levanta `ValueError("Status inválido.")`.
- Não há regra hoje impedindo "voltar" de `Em Atendimento` para `Aberto` —
  é uma decisão consciente para manter o piloto simples.

### 3.4 Status de SLA (`obter_metricas_sla`)
Calculado em runtime (não persistido): para chamados `Resolvido`, compara
`data_resolucao` com `prazo_limite`; para os demais, compara o `agora` com o
`prazo_limite`. Resultado: `"Dentro do SLA"` ou `"SLA Estourado"`.

### 3.5 Base de conhecimento — grava e busca

- `salvar_solucao_na_kb(problema, solucao, modulo_sap=None)`: faz
  `INSERT ... ON CONFLICT(problema_chave) DO UPDATE`, ou seja, se já existir
  uma entrada com a mesma chave normalizada, a solução é **sobrescrita**, o
  `contador_uso` é incrementado e `modulo_sap` só é atualizado se o novo
  valor não for nulo (mantém o antigo por `COALESCE`).
  - **[NOVO R3]** Gera automaticamente um embedding (vetor vetorial) da
    solução e armazena na coluna `embedding` para posterior busca semântica.
- `buscar_solucao_exata_kb(problema)`: normaliza o texto recebido e busca
  match exato na tabela. Se encontrar, incrementa `contador_uso` e retorna a
  solução; senão retorna `None`. **Não há fallback fuzzy/semântico** — dois
  textos parecidos mas não idênticos após normalização não batem.
- **[NOVO R3]** `buscar_solucao_similar_kb(problema, threshold=0.7)`: busca
  por solução *similar* (não idêntica) usando embedding vetorial. Compara o
  embedding do problema fornecido com todos os embeddings da KB e retorna a
  solução mais parecida **se** a similaridade cosseno > threshold (padrão 70%).
  Retorna um dict `{ "solucao": str, "confianca": float, "tipo": "similar" }`
  ou `None`.

### 3.6 Origem da solução
Sempre que um chamado é resolvido, `origem_solucao` registra de onde veio:
`Manual` (equipe digitou à mão), `Base de Conhecimento` (achou match exato na
KB) ou `IA` (Gemini gerou uma solução nova).

## 4. Contrato das funções — `backend.py`

> Convenção de erro: todas as funções de escrita/validação levantam
> `ValueError` com mensagem pronta para exibir ao usuário (`str(erro)`).
> Não há um sistema de códigos de erro — a interface apenas captura a
> exceção e mostra `st.error(str(erro))`.

### `init_db() -> None`
Cria as tabelas se não existirem, aplica migrações leves de coluna, e
garante que exista pelo menos uma norma ativa. **Deve ser chamada no início
de todo app** (`cliente_app.py`, `equipe_app.py`) e também no início de
qualquer fluxo do `aiengine.py` que dependa do schema.

### `criar_chamado(usuario, modulo_sap, descricao, prioridade="Média") -> int`
- Valida `usuario`, `modulo_sap`, `descricao` como não-vazios (`ValueError`
  caso contrário).
- Calcula `prazo_limite` via `calcular_horas_sla`.
- Insere e retorna o `id` gerado (`cursor.lastrowid`).

### `obter_chamado(ticket_id: int) -> dict | None`
Retorna o registro completo do ticket (via `dict_factory`) ou `None` se não
existir.

### `listar_chamados(status_filtro: str | None = None) -> list[dict]`
Lista todos os tickets, ordenados por `data_criacao DESC`. Se
`status_filtro` for informado, filtra por esse status. Usado pela **equipe**
(vê todos os chamados).

### `listar_chamados_por_usuario(usuario: str, status_filtro: str | None = None) -> list[dict]`
Igual ao anterior, mas filtrado por `usuario`. Usado pelo **cliente** (vê só
os próprios chamados).

### `atualizar_status_chamado(ticket_id: int, status: str) -> None`
Só aceita `Aberto` ou `Em Atendimento`. Levanta `ValueError` se o status for
inválido ou se o `ticket_id` não existir (`rowcount == 0`).

### `resolver_chamado(ticket_id, solucao, origem_solucao=ORIGEM_MANUAL) -> None`
- Valida que `solucao` não é vazia.
- Seta `status = Resolvido`, `data_resolucao = agora`, grava `solucao_aplicada`
  e `origem_solucao`.
- Levanta `ValueError` se o ticket não existir.
- **Não** salva automaticamente na KB — quem chama essa função decide se
  quer chamar `salvar_solucao_na_kb` em seguida (ver seção 3.5 e fluxo em
  `04-fluxos-e-integracao.md`).

### `salvar_solucao_na_kb(problema, solucao, modulo_sap=None) -> None`
Ver seção 3.5.

### `buscar_solucao_exata_kb(problema: str) -> str | None`
Ver seção 3.5.

### `buscar_solucao_similar_kb(problema: str, threshold=SIMILARIDADE_THRESHOLD) -> dict | None`
Busca por solução similar na KB usando embedding vetorial (busca semântica).
- Gera embedding do problema fornecido.
- Compara com todos os embeddings armazenados na KB.
- Retorna a solução mais parecida **se** a similaridade cosseno estiver acima do
  threshold (padrão: 0.7 = 70%).
- Retorna um dict com `{ "solucao": str, "confianca": float (0-1), "tipo": "similar" }`
  ou `None` se nenhuma atingir o threshold.
- Quando encontra, incrementa `contador_uso` da entrada, assim como o match exato.

### `listar_base_conhecimento() -> list[dict]`
Lista todos os registros da KB, ordenados por `contador_uso DESC,
data_atualizacao DESC` (mais usados e mais recentes primeiro).

### `listar_normas_modelo() -> list[dict]`
Retorna apenas `titulo` e `conteudo` das normas com `ativo = 1`, ordenadas
por `id`.

### `salvar_norma_modelo(titulo, conteudo) -> None`
Insere uma nova norma (sempre `ativo = 1` — não há UI hoje para desativar
normas antigas; ver roadmap).

### Estados e Transições (Máquina de Estados)

| Estado | Transições | Descrição |
|---|---|---|
| `Aberto` | → Em Atendimento, Aguardando confirmação, Resolvido | Chamado novo, não iniciado |
| `Em Atendimento` | → Resolvido, Aberto | Equipe está trabalhando |
| `Aguardando confirmação` **[NOVO R5]** | → Resolvido, Aberto | IA sugeriu; cliente confirma ou rejeita |
| `Resolvido` | (final) | Chamado fechado |

**Fluxo típico (cliente com IA)**:
- `Aberto` → `Aguardando confirmação` (IA sugeriu)
- `Aguardando confirmação` → `Resolvido` (cliente confirmou) ou `Aberto` (cliente rejeitou)

**Fluxo típico (equipe resolve)**:
- `Aberto` → `Em Atendimento` → `Resolvido`

### `aplicar_sugestao_ia(ticket_id, confirmou) -> None`
**[NOVO R5]** Cliente confirma ou rejeita a sugestão da IA.
- `confirmou=True`: move para `Resolvido`
- `confirmou=False`: volta para `Aberto` (retorna à fila da equipe)
Lê todos os tickets, calcula `status_sla`, `tempo_resolucao_horas` e
`horas_ate_prazo` como colunas derivadas (ver seção 3.4). Retorna
`pd.DataFrame()` vazio se não houver tickets — quem consome deve checar
`.empty` antes de usar.

### `notificar_equipe_sobre_divergencia_prioridade(ticket_id, prioridade_cliente, prioridade_sugerida, justificativa) -> None`
**[NOVO R4]** Notifica todos os usuários da equipe (role='equipe') sobre uma
divergência entre a prioridade escolhida pelo cliente e a sugerida pela IA.
Cria uma notificação de tipo `NOTIF_PRIORIDADE_DIVERGE` para cada membro da
equipe com a mensagem descrevendo a divergência e justificativa.

## 5. Contrato das funções — `aiengine.py`

### `obter_llm() -> ChatGoogleGenerativeAI`
Instancia o cliente do Gemini com `model=GEMINI_MODEL`, `temperature=0.2`,
sem limite de `max_tokens`, `max_retries=2`. A API key é lida
automaticamente da variável de ambiente `GOOGLE_API_KEY` (via
`load_dotenv()` no import do módulo).

### `montar_normas_modelo() -> str`
Busca as normas ativas via `backend.listar_normas_modelo()` e monta uma
string em bullet list (`- Título: Conteúdo`) para injetar no prompt do
sistema. Se não houver normas cadastradas, usa um texto padrão de fallback.

### `analisar_e_resolver_chamado(descricao_chamado, modulo_sap) -> str`
Fluxo:
1. `backend.init_db()` (idempotente, garante schema).
2. Tenta achar solução exata na KB via `buscar_solucao_exata_kb`. Se
   encontrar, retorna imediatamente prefixado com
   `"[Solução cadastrada encontrada]\n"` — **não chama o Gemini** nesse
   caso.
3. Se não encontrar, monta o prompt (system + human, ver seção 5.1),
   invoca a chain do LangChain, extrai `resposta.content.strip()`.
4. Salva a nova solução na KB via `salvar_solucao_na_kb`.
5. Retorna o texto da solução gerada.

> Convenção importante: o prefixo `"[Solução cadastrada encontrada]"` é
> usado por `resolver_ticket_com_ia` (abaixo) para decidir se a origem foi
> `ORIGEM_KB` ou `ORIGEM_IA`. Se alterar esse prefixo, atualizar os dois
> lugares.

#### 5.1 Estrutura do prompt

**System prompt** (resumo do conteúdo, não literal):
- Define o papel: especialista de suporte SAP.
- Regras: não inventar transações, não assumir dados ausentes, ser direto,
  indicar quando precisa de validação humana, responder em pt-BR.
- Injeta as normas internas ativas (via `montar_normas_modelo()`).

**Human prompt**:
- Módulo SAP afetado.
- Descrição do incidente.
- Pede resposta estruturada em 4 blocos: (1) Diagnóstico provável,
  (2) Passos recomendados, (3) Validações necessárias, (4) Quando escalar
  para um consultor SAP.

### `resolver_ticket_com_ia(ticket_id: int) -> str`
Função de mais alto nível, chamada diretamente pelas interfaces:
1. `backend.init_db()`.
2. Busca o chamado (`obter_chamado`); levanta `ValueError` se não existir.
3. Chama `analisar_e_resolver_chamado` com a descrição e módulo do chamado.
4. Decide `origem` (`ORIGEM_KB` se a resposta começar com o prefixo da KB,
   senão `ORIGEM_IA`).
5. Chama `resolver_chamado(ticket_id, solucao, origem)` — ou seja, **essa
   função já fecha o chamado como `Resolvido`**, não deixa em
   `Em Atendimento`.
6. Retorna o texto da solução para a interface exibir.

### [NOVO R6] Funções de Edição/Gestão da KB e Normas

#### `editar_solucao_kb(kb_id: int, nova_solucao: str, novo_modulo_sap: str | None = None) -> None`
Edita uma entrada existente da KB:
- Valida que `nova_solucao` não é vazia.
- Regenera automaticamente o embedding para a nova solução (para busca semântica funcionar).
- Atualiza `solucao_recomendada`, `embedding`, e opcionalmente `modulo_sap`.
- Atualiza `data_atualizacao` para o momento atual.
- Levanta `ValueError` se o `kb_id` não existir.

#### `excluir_entrada_kb(kb_id: int) -> None`
Deleta uma entrada da KB permanentemente.
- Levanta `ValueError` se o `kb_id` não existir.
- **Aviso**: é uma deleção hard (sem soft delete) — útil para limpar entradas erradas ou desatualizadas.

#### `desativar_norma(norma_id: int) -> None`
Desativa (seta `ativo = 0`) uma norma sem apagar o histórico.
- Normas desativadas não entram no prompt do Gemini (via `montar_normas_modelo()` que filtra só `ativo = 1`).
- Levanta `ValueError` se o `norma_id` não existir.

#### `reativar_norma(norma_id: int) -> None`
Reativa (seta `ativo = 1`) uma norma desativada.
- Levanta `ValueError` se o `norma_id` não existir.

#### `listar_todas_as_normas() -> list[dict]`
Lista **todas** as normas, ativas E desativadas, para exibição na UI de gerenciamento.
- Ordena por `ativo DESC, data_registro DESC` (ativas primeiro, depois desativadas).
- Diferente de `listar_normas_modelo()` que só retorna as ativas.

### [NOVO R7] Funções de Interações (Conversa dentro do Chamado)

#### `criar_interacao(ticket_id: int, usuario_id: int, mensagem: str) -> int`
Cria uma nova interação (comentário/mensagem) em um chamado:
- Valida que `mensagem` não é vazia.
- Verifica que `ticket_id` e `usuario_id` existem no banco.
- Insere em `ticket_interacoes`.
- Retorna o `id` da interação criada.
- Levanta `ValueError` se ticket ou usuário não existir.

#### `listar_interacoes(ticket_id: int) -> list[dict]`
Lista todas as interações de um chamado:
- Retorna lista com campos: `id, ticket_id, usuario_id, username, role, mensagem, data_criacao`.
- Ordena por `data_criacao ASC` (mais antigas primeiro).
- Inclui dados do usuário (nome e role) para exibição.

#### `contar_interacoes(ticket_id: int) -> int`
Conta quantas interações um chamado tem (usado para exibir total na UI).

## 6. Concorrência / integridade

- `get_connection()` abre a conexão SQLite com `timeout=10`,
  `PRAGMA journal_mode=WAL` e `PRAGMA busy_timeout=5000`. Isso permite que
  `cliente_app.py` e `equipe_app.py` leiam/escrevam quase simultaneamente
  sem erro de lock na maioria dos casos, mas **não** é uma solução para alta
  concorrência real (múltiplos atendentes resolvendo ao mesmo tempo em
  volume) — para isso, migrar para Postgres é o próximo passo natural (ver
  roadmap).
- Não há transações explícitas cobrindo múltiplas tabelas (ex.: resolver
  chamado + salvar na KB são duas chamadas/duas conexões separadas hoje).
  Se um IDE-AI for refatorar isso, vale considerar envolver em uma única
  transação para evitar estado inconsistente em caso de falha no meio do
  processo.

## 7. Fora de escopo deste backend (propositalmente)
- Autenticação/autorização.
- Anexos/upload de arquivos nos chamados.
- Histórico de comentários/interações dentro de um chamado (hoje só existe
  descrição inicial + solução final).
- Notificações (e-mail, Slack, Teams).
- Busca semântica na KB.
