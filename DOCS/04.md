# 04 — Fluxos e Integração

Este documento descreve, passo a passo, os principais fluxos ponta a ponta,
cruzando backend + frontend cliente + frontend equipe + IA.

---

## Fluxo 1 — Cliente abre chamado SEM acionar IA

1. Cliente preenche o formulário em `cliente_app.py` → aba "Abrir chamado".
2. `backend.criar_chamado(...)` grava na tabela `tickets` com
   `status = Aberto` e `prazo_limite` calculado pela prioridade.
3. Chamado fica visível:
   - Para o próprio cliente, em "Meus chamados" (via
     `listar_chamados_por_usuario`).
   - Para a equipe, na aba "Fila" de `equipe_app.py` (via `listar_chamados`),
     assim que ela atualizar a tela (manual ou pelo auto-refresh de 15s).
4. Equipe resolve manualmente (Fluxo 3) ou aciona a IA pela fila (Fluxo 4).

## Fluxo 2 — Cliente abre chamado COM IA acionada na hora **[MODIFICADO R5]**

1. Igual ao passo 1–2 do Fluxo 1.
2. Como o checkbox "Tentar resolução automática com IA agora" está marcado, 
   `cliente_app.py` chama `aiengine.resolver_ticket_com_ia(ticket_id)` 
   **imediatamente**.
3. **[NOVO R5]** Status é setado para `"Aguardando confirmação"` (não `Resolvido`)
4. Dentro dessa chamada (ver `01-backend-especificacao.md` §5):
   - Se já existir uma solução exata na KB para aquela descrição
     normalizada → retorna na hora, sem chamar o Gemini, marca
     `origem_solucao = Base de Conhecimento`.
   - Se não existir → chama o Gemini com o prompt + normas ativas, salva a
     resposta na KB, marca `origem_solucao = IA`.
5. **[NOVO R5]** Cliente vê a sugestão com **dois botões de confirmação**:
   - `"✓ Resolveu meu problema!"` → move para `Resolvido` (chamado fecha)
   - `"✗ Não resolveu, quero falar com a equipe"` → volta para `Aberto` 
     (equipe pode pegar da fila)
6. **[NOVO R5]** Se rejeitar, notifica equipe que cliente não aceitou a 
   sugestão (pode incluir nova notificação de tipo "sugestao_rejeitada")

## Fluxo 3 — Equipe resolve manualmente

1. Equipe encontra o chamado na aba "Fila", digita o ID no
   `number_input`, escreve a solução no expander "Resolver manualmente".
2. `backend.resolver_chamado(id, solucao, origem_solucao=Manual)` → status
   vira `Resolvido`.
3. Em seguida, `equipe_app.py` chama
   `backend.salvar_solucao_na_kb(problema=descricao_original, solucao, modulo_sap)`
   — a solução manual **também alimenta a base de conhecimento**, para que
   da próxima vez que alguém relatar o mesmo problema (mesma descrição
   normalizada), a IA/KB já devolva essa solução sem gastar chamada ao
   Gemini.
4. Cliente vê a solução assim que atualizar "Meus chamados".

## Fluxo 4 — Equipe aciona a IA pela fila
Mesmo mecanismo do passo 3 do Fluxo 2, só que disparado pela equipe em vez
do cliente (botão "Resolver com IA" na aba Fila). Resultado idêntico:
chamado fechado, solução salva/reaproveitada na KB.

## Fluxo 5 — Reaproveitamento de solução (o "diferencial IA" do projeto)

### A. Reaproveitamento exato (mesmo texto, normalizado)
1. Usuário A relata um problema, descrição X. IA gera solução, que é salva
   na KB com chave = `normalizar_texto(X)` + seu embedding.
2. Usuário B relata **exatamente o mesmo texto** (ou texto que, depois de
   normalizado — minúsculas, espaços colapsados — fica idêntico a X).
3. Ao acionar IA (cliente ou equipe), `buscar_solucao_exata_kb` encontra o
   match exato, retorna a solução salva **sem chamar o Gemini de novo**, e
   marca `origem_solucao = Base de Conhecimento`.

### B. Reaproveitamento similar (texto parecido, não idêntico)
**[NOVO R3]**
1. Usuário C relata um problema **parecido** com X, mas com palavras
   diferentes. A descrição normalizada não é idêntica, então o match exato
   falha.
2. Ao acionar IA, `buscar_solucao_similar_kb` gera embedding da descrição
   de C, compara com todos os embeddings da KB (similaridade cosseno), e
   encontra o problema de A com confiança 85% (acima do threshold 70%).
3. Retorna a solução de A, marcando `origem_solucao = Base de Conhecimento`
   e exibindo na interface: "≈ Solução similar (confiança: 85%)".

### C. Limite conhecido (ainda não resolvido)
Se o usuário D escrever o mesmo problema com palavras muito diferentes (ex.
slangs, sinônimos não óbvios), a similaridade pode ficar abaixo do threshold
e cair no Gemini. Isso gera uma **nova entrada na KB** em vez de reaproveitar.
Isso é **esperado no piloto** (matching por embeddings, não NLP semântico
avançado com contexto de domínio). Ver R11 (fine-tuning com histórico real)
se isso virar um problema em produção.

## Fluxo 6 — Como as duas interfaces se "sincronizam"

- Não existe push/notificação entre os dois apps — cada um é um processo
  Streamlit independente.
- A "sincronização" é indireta: os dois leem o mesmo arquivo
  `sap_chamados.db`, então qualquer escrita feita por um fica visível para
  o outro assim que ele reler o banco.
- Cada app relê o banco quando:
  - Há qualquer interação do usuário que dispare um rerun (clique de
    botão, mudança de campo, submit de formulário);
  - O timer de auto-refresh (`streamlit_autorefresh`, 15s) dispara — se a
    lib estiver instalada;
  - O botão manual "🔄 Atualizar" é clicado.
- Concorrência de escrita é mitigada por `PRAGMA journal_mode=WAL` +
  `busy_timeout=5000` no `backend.py` (ver `01-backend-especificacao.md`
  §6), mas não há fila/lock de aplicação — é possível, em teoria, dois
  processos tentarem resolver o mesmo ticket ao mesmo tempo e um sobrescrever
  o outro silenciosamente (last-write-wins). Para o volume de um piloto
  isso é aceitável; não é para produção com múltiplos atendentes
  simultâneos em volume alto.

## Fluxo 7 — Conversa/Interações dentro do Chamado **[NOVO R7]**

Permite que cliente e equipe se comuniquem **durante** o atendimento, criando
um histórico de mensagens dentro do chamado:

1. **Cliente envia mensagem**: em "Meus chamados", seção "Conversa", cliente
   digita uma mensagem → chama `backend.criar_interacao(ticket_id, usuario_id, mensagem)`.
2. **Equipe vê mensagem**: quando equipe atualizar/recarregar a fila ou
   abrir o chamado, a mensagem aparece na seção "Conversa" com emoji 👤 e
   nome do cliente.
3. **Equipe responde**: na aba "Fila", seção "Conversa" do chamado, equipe
   digita uma resposta → chama `backend.criar_interacao(ticket_id, usuario_id, mensagem)`.
4. **Cliente vê resposta**: na próxima atualização de "Meus chamados",
   mensagem da equipe aparece com emoji 👨‍💼 e nome do atendente.

Fluxo é **assíncrono** (não há notificação em tempo real):
- Mensagens são lidas via `backend.listar_interacoes(ticket_id)` a cada rerun/atualização.
- Ordem cronológica: `ORDER BY data_criacao ASC` (mais antigas primeiro).
- Útil para esclarecimentos, acompanhamento, feedback antes de resolver.

## Fluxo 8 — Inicialização do banco
Tanto `cliente_app.py` quanto `equipe_app.py` chamam `backend.init_db()`
logo no início da execução (é idempotente — pode rodar múltiplas vezes sem
efeito colateral). Isso significa que **o primeiro app que subir cria o
banco e a norma padrão**; o segundo app, ao subir, só confirma que o schema
já existe.


## Fluxo 9 — Upload de Anexos **[NOVO R10]**

### Cliente anexa arquivo:
1. Cliente seleciona chamado em "Meus chamados"
2. Na seção "📎 Anexos", clica em "Adicionar anexo"
3. Seleciona arquivo (JPG, PNG, PDF, DOC, XLS, TXT)
4. Sistema salva arquivo em `anexos/{uuid}_{nome}`
5. Registra metadados em `ticket_anexos`
6. Arquivo fica disponível para download imediato

### Equipe visualiza/baixa anexo:
1. Equipe seleciona chamado na fila
2. Na seção "📎 Anexos", vê lista de arquivos
3. Clica em botão "⬇️" para download
4. Arquivo baixado com nome original

### Considerações:
- Arquivos armazenados localmente (fácil migração para S3/cloud)
- Tamanho limitado pela configuração Streamlit (200MB padrão)
- Tipos de arquivo restritos por segurança
- Backup deve incluir pasta `anexos/`

## Fluxo 10 — Migração para PostgreSQL **[NOVO R9]**

### Configuração inicial:
1. Administrador instala PostgreSQL
2. Configura `.env` com credenciais: `DB_TYPE=postgres`, `PG_HOST`, etc.
3. Executa script de migração (`pgloader` ou manual)
4. Testa conexão e funcionalidades

### Operação diária:
1. Ambos apps (`cliente_app.py`, `equipe_app.py`) usam `backend.get_connection()`
2. Função detecta `DB_TYPE` e conecta ao banco apropriado
3. Queries funcionam igual em ambos bancos (compatibilidade de schema)
4. Performance melhorada para múltiplos usuários simultâneos

### Backup/Recovery:
1. PostgreSQL: `pg_dump` automático (cron job)
2. SQLite: cópia manual do arquivo `.db`
3. Migração reversível alterando `DB_TYPE`

## Fluxo 11 — Análise para Treinamento de Modelo Próprio **[NOVO R11]**

### Análise de viabilidade:
1. Administrador acessa aba "🤖 ML" no painel da equipe
2. Configura custo estimado do Gemini por chamada
3. Executa análise automática
4. Recebe recomendação (ALTO/MODERADO/BAIXO) com justificativa

### Exportação para treinamento:
1. Se recomendação = "ALTO", clica em "Exportar Dataset"
2. Sistema exporta `tickets` + `knowledge_base` como JSONL/CSV/Parquet
3. Gera relatório com estatísticas e análise
4. Arquivos salvos em `data/training/`

### Preparação para ML:
1. Configurar ambiente Databricks/MLflow (guia em `databricks_training.py`)
2. Implementar pipeline real de treinamento (esqueleto fornecido)
3. Treinar modelo com dados exportados
4. Validar vs. Gemini (métricas: BLEU, ROUGE, Precision@K)
5. Deploy e A/B testing

### Considerações:
- Implementação atual é POC/esqueleto
- Requer integração real com ambiente ML em produção
- Volume mínimo recomendado: 500+ chamados resolvidos
- ROI positivo normalmente a partir de 1000+ chamados/mês
